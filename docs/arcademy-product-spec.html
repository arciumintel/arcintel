Product & engineering spec

Arcademy

Arcademy is onboarding infrastructure for the Arcium ecosystem. It is not a school platform or a generic course website.

Partners turn scattered documentation into a guided program: ordered lessons, short understanding checks, and tracked progress. Users get one account, a curated catalog, and proof they actually understood the material—not just that they scrolled past it.

One sentence: Arcademy helps ecosystem teams onboard new users and developers with guided programs, comprehension checks, and measurable progress—on shared infrastructure instead of everyone building their own onboarding from scratch.
First program at launch: Arcium Fundamentals Single public catalog (one hub) Staff approves everything that goes live in v1

Who uses Arcademy

Learners

New users and developers exploring Arcium ecosystem products.

  • Browse programs on the hub
  • Read lessons in order
  • Pass short quizzes to confirm understanding
  • One account; progress saved per program
  • Try lesson 1 as a guest, then sign up to keep progress (Phase 1)

Partners

Ecosystem teams (protocols, apps, tools) who need users to get productive—not just read docs.

  • Submit an intake brief; staff builds the first version
  • Later: edit drafts in Partner Studio after earning trust
  • See where users drop off (Phase 2+)
  • Cannot go live without staff approval in v1

Staff

Arcademy operators who run the hub and onboard partners.

  • Create organizations and programs
  • Write, review, and publish content
  • Choose what is listed or featured on the hub
  • Grant trusted-partner status after quality review

Core ideas (plain language)

Term What it means for developers
Hub Public home and catalog: / and /programs. Only programs staff mark as listed or featured appear here.
Program One onboarding path (e.g. Arcium Fundamentals). Users enroll in a program; progress is scoped to that program.
Organization The partner company that owns programs. Used for permissions and keeping each partner’s data separate.
Lesson A reading-first page built from fixed block types (heading, paragraph, callout, code, image, divider)—not free-form HTML.
Quiz Optional check at the end of a lesson. Question shapes are defined by the platform. Answer keys stay on the server.
Draft vs published Authors edit a working draft. Publish creates a frozen snapshot. Never edit published snapshots in place.
Enrollment A user’s membership in one program, pinned to a specific published curriculum version.
Tenant isolation Partner A cannot read Partner B’s drafts or user progress. Enforced in the database and in application code.

How it works

Learner journey

Discover → open program → read lesson → pass check → progress saved → program complete

Content lifecycle (staff / trusted partner)

Publishing and hub listing are separate steps. A program can be published but still hidden from the catalog.

Data shape (simplified)

Each partner org owns programs. Published content is versioned; progress points at lesson/quiz versions, not mutable drafts.

Pages developers build

Build in phase order. Do not ship public learner pages before the database foundation and isolation tests are done.

Route Who Purpose Phase
/ Everyone Hub home — featured programs, continue learning 1
/programs Everyone Full catalog (listed / featured only) 1
/programs/[slug] Everyone Program overview, enroll or continue 1
/programs/[slug]/lessons/[lesson] Everyone Lesson reader + quiz 1
/account Signed-in users Enrollments and progress across programs 1
/staff/* Staff Authoring, publish, hub curation, intake 2
/partner/* Trusted partners Draft editing and submit for review (no self-publish) 3

Old /modules/* URLs should redirect to /programs/arcium/* for bookmarks and search engines (Phase 0).

Build now — Phase 0 (foundation)

Do this first Phase 0 has no public hub UI. Goal: multi-tenant database, isolation, Arcium content seeded, and all reads/writes through scoped repositories.

Milestone 1–3 · Multi-tenant core, migration, verification

~5 weeks · Backend-heavy
  • ARC-5 Tenancy schema — organizations, programs, curriculum versions, lessons, quizzes, enrollments, progress tables. Run via npm run db:migrate.
  • ARC-6 Row-level security — database policies so partners cannot see each other’s data. Session variables set per request.
  • ARC-8 Tenant context + repositories — resolveTenantContext(); no raw database queries in route handlers.
  • ARC-7 Connection spike — confirm Neon unpooled connections work with row-level security (documented in agent platform doc).
  • ARC-9 Program-scoped content layer — refactor reads to use program + version, not legacy module paths.
  • ARC-10 Migrate Arcium — first program arcium with curriculum version 1 and lesson snapshots.
  • ARC-11 Pin existing learners — backfill enrollments and progress with version foreign keys.
  • ARC-12 Legacy redirects — /modules/* → /programs/arcium/*.
  • ARC-13 Lesson block schema v1 — validate and render structured blocks only.
  • ARC-14 Cross-tenant tests — user A must get 403/404 for partner B data, never an empty 200.
Phase 0 exit: Arcium content is readable through repositories; isolation tests pass. Learner-facing hub pages start in Phase 1.

Do not in Phase 0

  • Ship /programs public UI before repositories and row-level security work
  • Build staff authoring UI (use seed scripts; UI comes Phase 2)
  • Let partners publish to production without staff
  • Update published lesson rows in place
  • Send quiz answer keys to the browser

Build next — in chronological order

After Phase 0 passes its exit criteria, work through these phases in order. Do not skip ahead.

Phase 1 · Hub and learner loop

M4–M5 · ~4 weeks

First user-visible Arcademy experience. Arcium appears on the hub.

  • ARC-15–18Hub home, program catalog, program page, lesson player
  • ARC-20Guest can try lesson 1; progress stored locally until signup
  • ARC-19Global account enrollments (decide: enroll button vs first activity)
  • ARC-21Merge guest progress when user creates account
  • ARC-22Platform events (enrolled, lesson started/completed, quiz passed) in same transaction as progress writes
  • ARC-23Staff UI to set listed / featured / sunset on hub
Exit: Visitor completes Arcium lesson 1 as guest, signs up, progress is kept. Hub shows Arcium.

Phase 2 · Staff authoring and second program

M6–M7 (part 1) · ~4 weeks
  • ARC-26Draft → review → approved → published state machine
  • ARC-25Staff Studio scoped by organization and program
  • ARC-27Publish and rollback UI (new immutable version each publish)
  • ARC-24Partner intake form → staff queue → create org/program
  • ARC-28Signed preview links for draft curriculum
  • ARC-29–30Nightly analytics rollups + staff funnel dashboard
  • ARC-31Second program built end-to-end by staff (first non-Arcium pilot)
Exit: A second partner program is live on the hub, built entirely through staff tools.

Phase 3 · Trusted partners

M7 (part 2) · ~2 weeks
  • ARC-34Staff sets organization trust level after QA
  • ARC-32Partner Studio — draft only, submit for review
  • ARC-33Staff approval queue
  • ARC-35–36Partner read-only analytics; gamification preset toggles (minimal vs standard)
  • ARC-37Trusted partner pilot — partner submits draft, staff publishes
Exit: Trusted partner ships a curriculum update without staff rewriting from scratch.

Phase 4 · Later (post–v1)

Deferred issues ARC-38–44

Not required for first public launch. Planned when product and legal gates clear:

  • Signed completion records partners can verify
  • Webhooks for partner apps
  • Sign-in through partner apps
  • Embeddable widget
  • More lesson block types
  • Automated enrollment migration between curriculum versions

Dev guardrails

Rule Why
All server data access goes through resolveTenantContext() and repositories Consistent permissions; easier audits
Publish = copy draft into new version rows Learners always see stable content; progress stays valid
Quiz scoring only on server Prevents cheating; answer keys never in client bundles
Lessons use block schema (Zod), not author HTML Safer rendering; consistent UX
Content lives in Postgres at runtime No parallel “lessons in git” source of truth
Every migration hits production database Preview/dev/prod share DB until infra splits—migrate carefully

Stack: Next.js 16, React 19, TypeScript, Tailwind, PostgreSQL (Neon), better-auth, Zod, Cloudinary for images, Vercel hosting.

In v1 / not in v1

In v1

  • Hub and program catalog
  • Lesson player with platform quizzes
  • One global Arcademy account
  • Staff-built partner programs
  • Arcium as first program
  • Guest try → sign up → keep progress
  • Staff analytics on drop-off

Not in v1

  • Generic school / classroom features
  • Custom quiz question types
  • Public leaderboards, streaks, points
  • Partner self-publish without staff
  • Embeds in partner apps
  • Sign-in through partner apps
  • Billing and webhooks

Future direction

  • Signed completion records
  • Named credentials (“finished Arcium Fundamentals”)
  • Exportable progress history
  • Not a token-rewards game—proof of learning matters

Decisions already made

  • Product category: onboarding infrastructure, not an education platform
  • Single public hub (not separate sites per partner)
  • Arcium is program #1 at launch (arcium slug)
  • Staff must approve all publishes in v1
  • Published snapshots are immutable
  • English-only UI for v1
  • Fewer than ~10 partner programs expected in year one
